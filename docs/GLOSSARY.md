# Organizational Schema Theory: Glossary

Key terms used throughout the prompt toolkit.

---

## Cascade Levels

| Term | Level | TDD Analogy | Description |
|:-----|:------|:------------|:------------|
| Customer Experience Contract | L0 | Acceptance test | What customers should perceive and feel |
| Signal Requirement | L1 | Integration test | What signals produce that perception |
| Process Contract | L2 | Unit test | What processes must achieve (measurable) |
| Procedure | L3 | Implementation | How an executor achieves a contract |
| Input Specification | L4 | Dependency | Materials, equipment, capabilities required |
| Sourcing Requirement | L5 | Infrastructure | Where inputs come from |

## Contract Types (L0)

| Type | Description | Example |
|:-----|:------------|:--------|
| Experience | Customer-derived perception goal | "Espresso is balanced, sweet, clean finish" |
| Constraint | Regulatory or legal requirement | EU Regulation 852/2004 food safety |
| Commitment | Voluntary organizational promise | "All coffee from direct trade relationships" |

## Linking Conventions

| Field | Direction | Meaning |
|:------|:----------|:--------|
| `satisfies_experience` | L1 -> L0 | Signal produces this experience |
| `satisfies_signal` | L2 -> L1 | Process emits this signal |
| `satisfies_constraint` | L2 -> L0 | Process meets this regulation |
| `satisfies_commitment` | L2 -> L0 | Process fulfills this commitment |
| `implements_contract` | L3 -> L2 | Procedure achieves this contract |
| `requires_input` | L4 -> L3 | Input needed for this procedure |
| `sourced_from` | L5 -> L4 | Supplier provides this input |

## Executor Types

| Type | Description | Signal Behavior |
|:-----|:------------|:----------------|
| Human | Human operator following procedures | Reprocesses signals through personal priors; emits secondary analogue signals |
| Hybrid | Human with automated assistance | Partial reprocessing; automation handles precision tasks |
| Automated | Fully automated system | Reproduces signals faithfully; no secondary signal emission |

## Signal Concepts

| Term | Definition |
|:-----|:-----------|
| Designed signal | Intentionally created by the business (e.g., brand materials, trained greeting) |
| Ambient signal | Generated by the environment without explicit design (e.g., reviews, word-of-mouth) |
| Emission type | Whether a signal is designed or ambient |
| Signal dimension | Which of the 8 SBT perceptual dimensions a signal activates |
| Proxy indicator | A measurable metric that serves as a proxy for a subjective experience goal |

## SBT Dimensions

The eight perceptual dimensions from Spectral Brand Theory, used in L1 signal mapping:

| Dimension | What It Captures |
|:----------|:----------------|
| Semiotic | Visual and symbolic identity (logo, colors, typography, packaging) |
| Narrative | Stories and mythology (origin story, founder myth, brand history) |
| Ideological | Values, ethics, and beliefs (sustainability, fairness, transparency) |
| Experiential | Direct sensory and functional experience (taste, touch, speed, comfort) |
| Social | Interpersonal and tribal dynamics (staff interaction, community, belonging) |
| Economic | Pricing and value perception (premium positioning, cost-value ratio) |
| Cultural | Aesthetic codes and references (design sensibility, local adaptation, zeitgeist) |
| Temporal | Time-based patterns (innovation cadence, consistency, heritage) |

## Methodology Concepts

| Term | Definition |
|:-----|:-----------|
| TDD cascade | The six-level test hierarchy (L0-L5) |
| Contract | What must be achieved (the test) |
| Procedure | How it is achieved (the implementation) |
| Quality gate | A measurable parameter with min/max/target that a process must satisfy |
| Failure action | What happens when a quality gate is not met |
| Executor profile | A configuration specifying how a procedure is executed (human, hybrid, automated) |
| Active executor | The currently selected executor profile for a procedure |
| Forkability | The ability to copy a test suite and rewrite the implementation for a new context |
| Schema | What parameters exist (publishable) |
| Data | Specific parameter values (competitive moat) |
| Backward traceability | Every parameter traces to the L0 contract it serves |
| Orphan | A process, input, or supplier that does not trace to any L0 contract |
| Muda | Waste -- operational elements that do not contribute to customer experience |
| Starter pack | Pre-filled L0 contracts for a common business category |

## Validation Concepts

| Term | Definition |
|:-----|:-----------|
| Completeness | Every L0 contract has satisfying signals; every signal has supporting processes |
| Traceability | Every L5 requirement traces back to an L0 contract through the cascade |
| Consistency | No conflicting quality gates or contradictory constraints |
| Signal coverage | All 8 SBT dimensions addressed (warnings for gaps, not errors) |
| Waste detection | Identifying processes/inputs/suppliers that do not trace to L0 contracts |
